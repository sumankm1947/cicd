00Overview

This repository is a CI/CD learning lab, not a product. The author already knows API & UI test automation; the deliberate gap being closed here is continuous integration / delivery. A tiny Flask calculator + pytest suite exist only to give a pipeline something real to run, lint, test, containerise and "deploy."

The mental shift

Automation you already write (API/UI tests) answers "does the code work?". CI/CD answers "run that automatically on every change, block the merge if it's red, and ship only when it's green and approved."

How to read this guide

Each lab has a Goal, the key concept, a trimmed code example, the gotchas that bite in interviews, and a one-line soundbite you can recite. Skim the soundbites for a 5-minute refresh.

★ One-sentence summary

I built a 5-job GitHub Actions pipeline — lint → matrix tests → Docker build + DB integration → gated deploy — and progressively layered on matrices, caching, secrets, scheduling, services, environments, reusable workflows and HTML reporting.

Gotcha that recurs The API binds to host="0.0.0.0", not 127.0.0.1. Inside a Docker container 127.0.0.1 only listens inside the container — the CI smoke test (and your machine) could never reach it. This single line is what makes Lab 7's container reachable.

The first real bug: pythonpath in pytest.ini

SymptomModuleNotFoundError: No module named 'app' when running bare pytest — and it would fail in CI too.
Causepython -m pytest adds the current dir to sys.path; the bare pytest console script (what CI uses) does not.
Fixpythonpath = . in pytest.ini (pytest ≥ 7) — puts the project root on the import path no matter how pytest is invoked.
★ SDET soundbite

"How a tool is invoked changes its runtime behaviour — the classic 'passes on my machine, fails in CI' trap. Pinning it in config is the durable fix."

02The 6 core words

Get these exact and you can talk pipelines fluently.

TermWhat it is
WorkflowOne YAML file in .github/workflows/, triggered by events.
Event / triggerWhat starts a run: push, pull_request, schedule, manual workflow_dispatch. Declared under on:.
JobRuns on its own fresh, isolated VM (a runner). Jobs run in parallel by default.
StepOne ordered action in a job: either uses: (prebuilt action) or run: (shell).
ActionA reusable step from the Marketplace, e.g. actions/checkout@v4.
ArtifactA file a job produces (report, build) saved off the VM so it survives and can be downloaded.
hierarchyworkflow (ci.yml)
└── jobs
    └── test            (a JOB — its own VM)
        └── steps       (ordered commands)
            ├── checkout
            ├── setup-python
            └── pytest
The recurring theme Each job is a brand-new, isolated VM. Nothing carries between jobs — that's why every job repeats checkout/setup, and why sharing a file between jobs needs the artifact handoff. This one fact explains half the design decisions below.

L1Baseline pipeline

✓ donecheckout · setup · lint · test · artifact

Goal: one job that checks out the code, installs deps, lints, runs tests, and uploads machine-readable reports — proving the merge gate works end to end.

yaml · ci.yml (Lab 1 shape)on:
  push: { branches: [main] }
  pull_request:            # the SDET merge gate
  workflow_dispatch:       # manual "Run workflow" button
jobs:
  test:
    runs-on: ubuntu-latest   # Linux = fastest & cheapest runner
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.10", cache: pip }
      - run: pip install -r requirements.txt
      - run: flake8 app tests --select=E9,F63,F7,F82   # real errors fail
      - run: pytest --cov=app --cov-report=xml --junitxml=report.xml
      - uses: actions/upload-artifact@v4
        if: always()            # upload even when tests fail
        with: { name: test-reports, path: report.xml }

Artifacts — the key step

The pytest flags produce two machine reports: report.xml (JUnit — every test, pass/fail, duration) and coverage.xml (Cobertura — which lines ran). The VM is destroyed at job end, so upload-artifact saves them off-VM.

The one line that matters if: always() — by default a step is skipped after an earlier failure, but you want the report most when tests fail. always() forces the upload regardless. (Relatives: success(), failure(), cancelled().)
★ Soundbite

The JUnit/Cobertura XML is the raw material for everything downstream — PR test summaries, coverage gates, HTML reports. Lab 1 lays that foundation.

L2Matrix builds

✓ donestrategy.matrix · fail-fast · artifact naming

Goal: run the suite across Python 3.9–3.12 in parallel without writing four jobs.

yamltest:
  runs-on: ubuntu-latest
  strategy:
    fail-fast: false                # see gotcha 1
    matrix:
      python-version: ["3.9", "3.10", "3.11", "3.12"]
  steps:
    - uses: actions/setup-python@v5
      with: { python-version: ${{ matrix.python-version }} }
    - uses: actions/upload-artifact@v4
      with: { name: test-reports-${{ matrix.python-version }} }  # gotcha 2
Mental model strategy.matrix is job-level = "make N copies of this job." with: is step-level = "inputs to this one action." The matrix lives where the cloning happens; steps just read from it via ${{ matrix.x }}.
Gotcha 1 · fail-fastDefault true cancels sibling jobs the moment one goes red. As an SDET you want to see every version that breaks → set fail-fast: false.
Gotcha 2 · 409 collisionIn upload-artifact@v4 two artifacts in one run cannot share a name (v3 silently merged). Make it unique per clone: test-reports-${{ matrix.python-version }}.
★ Soundbite

One job definition → four parallel runners. That's how you prove cross-version support cheaply.

L3Multiple jobs & needs

✓ donejob graph · needs · fresh-VM isolation

Goal: split into a lint job and a test job, and make test wait for lint.

yamljobs:
  lint:                       # fast, cheap, once on 3.10
    runs-on: ubuntu-latest
    steps: [ checkout, setup-python, install, flake8 ]
  test:
    needs: lint                # <-- the dependency edge
    strategy: { matrix: { python-version: [...] } }
    steps: [ checkout, setup-python, install, pytest ]  # repeated on purpose
Actions ▸ run #42 ▸ lint error pushed
✓ lint red ✗ (style error)
– test (3.9) skipped
– test (3.10) skipped
– test (3.11) skipped
– test (3.12) skipped

Fig 2 — With needs: lint, a lint failure makes all four test jobs show skipped — fail-fast at the job level, before you burn four test runners.

Why the repetition isn't wasteEach job is a clean VM, so lint and test each repeat checkout + setup + install. That repetition is the isolation that makes jobs independent — and exactly why sharing a built file later needs the artifact handoff.
Lint runs onceLinting is Python-version-independent, so it's a single job on 3.10. Only test carries the matrix.
★ Soundbite

needs: draws the dependency edge in the job graph. Drop it and everything runs in parallel; add it and you get ordered, gated stages.

L5Caching, the explicit way

✓ doneactions/cache · keys · restore-keys

Goal: rip off the convenience cache: pip and wire the cache by hand, exposing the three things interviewers ask about: the key, the hit/miss, and the restore-keys fallback.

yaml- uses: actions/setup-python@v5
  with: { python-version: "3.10" }   # note: NO `cache: pip`
- uses: actions/cache@v4
  with:
    path: ~/.cache/pip                # cache the DOWNLOADS, not site-packages
    key: ${{ runner.os }}-pip-3.10-${{ hashFiles('requirements.txt') }}
    restore-keys: |
      ${{ runner.os }}-pip-3.10-     # prefix fallback (miss only)
- run: pip install -r requirements.txt  # HIT -> installs from cache

The two-part lifecycle people miss

  • Restore happens at the step — the runner looks for a matching cache and pulls it down.
  • Save happens automatically at job end (a hidden "post" step), and only on a miss. You never write a save step. Place the cache step before install.
The key is the whole lessonhashFiles('requirements.txt') fingerprints the deps. Edit the file → key changes → fresh cache. Same deps → same key → reused. restore-keys is a prefix list used only on a miss: grab the newest cache starting with the prefix, so a single dep bump downloads just the one changed package.
Matrix twistIn the matrix test job, put matrix.python-version in the key — otherwise all four versions fight over one cache and restore 3.9 wheels into a 3.12 run.
The one dangerCaching's failure mode is staleness — restoring old deps that should have changed. The content hash in the key prevents it. A cache key with no content hash is a bug.
★ Soundbite

cache: pip = "cache, just handle it." actions/cache = "here's exactly what to save, under what name, with what fallback." Reach for the explicit form when the default doesn't fit.

L6Scheduled & path-filtered runs

✓ donecron · paths · required-check deadlock

Goal: two triggers, same theme — not everything should run on every event.

Part 1 — nightly cron (in ci.yml)

yamlon:
  schedule:
    - cron: "0 3 * * *"   # 03:00 UTC daily

Fields: minute · hour · day-of-month · month · day-of-week.

Part 2 — path filter (app-only.yml)

yamlon:
  push:
    paths:
      - "app/**"          # fires only on app/ changes

Cron gotchas (interview-grade)

UTC onlyNo timezone field. Convert in your head.
Default branch onlyschedule runs only on main, using the workflow as it exists on main. Can't schedule a feature branch.
Best-effort timingDelayed under load, worst on the hour. Offset from 0 if timing matters.
Auto-disabledPaused after 60 days of repo inactivity until the next push.
The big trap — "skipped required check" deadlockIf a path-filtered workflow is a required status check, a PR that doesn't touch those paths shows it as skipped — and a required check that never reports passed can block the PR forever. Fix with a path-aware pattern or an always-green placeholder job.
Why a separate fileBolting paths: onto the main ci.yml push trigger would stop the real test suite from running on test-only/config commits — the wrong call for a gate. An isolated workflow demos path filters without weakening CI.
★ Soundbite

Schedules drive nightly regression / smoke suites; path filters skip work irrelevant to a change. Both say: don't run everything every time.

L7Services / Docker

✓ doneDockerfile · layer caching · services · health checks

Goal: two mechanisms — package the app as an image and build it in CI (real), and stand up a dependency container with services: (mechanism demo).

Docker vocabulary

ImageThe frozen, built package (the Dockerfile's output). Like an .exe on disk.
ContainerA running instance of an image. One image → many containers (class → objects).
LayerEach Dockerfile line is a cached layer. Identical base layers are stored once and shared.

Part B — the Dockerfile (deps before code)

dockerfileFROM python:3.10-slim
WORKDIR /app
COPY requirements.txt .                       # copy deps list FIRST...
RUN pip install --no-cache-dir -r requirements.txt  # ...this layer caches
COPY . .                                      # then copy app code
EXPOSE 5000
CMD ["python", "-m", "app.api"]
The key ideaDeps-before-code is the same "cache the deps, not the app" lesson as Lab 5, one level up. Change app code and Docker reuses the cached pip install layer instead of re-downloading everything.
.dockerignore trapUnlike .gitignore, .dockerignore does not support inline comments — a trailing # comment corrupts the pattern (Docker errors with "non-printable ASCII characters"). Comments go on their own line.

The CI job & the local-first loop

bash · local loopdocker build -t cicd-lab .
docker run --rm -p 5000:5000 cicd-lab
curl http://localhost:5000/health    # {"status":"ok"}

CI mirrors this: docker build → run detached (-d) → curl --fail /health (non-zero on HTTP error → fails the job) → clean up with if: always().

Part A — services: (the mechanism demo)

yamlintegration-db:
  services:                       # JOB-level key (sibling of runs-on/steps)
    postgres:                     # service id = its network hostname
      image: postgres:16
      options: >-                  # GitHub waits for healthy BEFORE steps run
        --health-cmd pg_isready --health-interval 10s ...
  steps:
    - run: psql -h localhost -U testuser -d testdb -c "SELECT version();"
Health check = no guessingGitHub won't start your steps until pg_isready reports healthy — the proper fix for the sleep 3 readiness race in Part B. No flake.
Addressing itSteps run on the runner VM, so connect at localhost:5432. If the job itself ran inside a container, you'd use the service name postgres as the host — a classic interview distinction.
★ Soundbite

Build the image in CI to catch "builds but app is broken"; use services: with health checks to give integration tests a real, ready dependency.

L8Environments & deploy gates

✓ doneCI → CD · environment · required reviewer

Goal: cross from CI (verify) to CD (ship) — with a human approval gate in front of the deploy.

yamldeploy:
  needs: test                          # 1) only after CI is green
  if: github.ref == 'refs/heads/main'   # 2) only from main, never PRs
  runs-on: ubuntu-latest
  environment: production            # 3) pause for a required reviewer
  steps:
    - run: echo "Deploying... (real deploy command here)"
Actions ▸ push to main ▸ awaiting approval
✓ lint 1m 02s
✓ test (×4) 2m 18s
✓ docker-build 1m 40s
⏸ deploy —  Review pending · Approve to continue waiting

Fig 4 — With environment: production + a required reviewer, the run pauses at deploy until a human clicks Approve.

The three stacked guards ARE the lessonneeds: test (green only) · if: …main (trunk only) · environment: production (human sign-off). Real release pipelines want all three.
One-time UI stepYAML can't create the environment. Do it at Settings → Environments → production and add yourself as Required reviewer. The gate fires on the event the job runs on — so you see the pause after merge to main, not on the PR.
★ Soundbite

The real release shape: tests pass → trunk only → human sign-off → ship. That's the merge gate plus the deploy gate.

L9Reusable & composite workflows

✓ doneworkflow_call · composite action · DRY

Goal: stop copy-pasting the setup block. GitHub gives two reuse mechanisms — I built both and composed them.

The distinction (the interview answer)

Composite actionReusable workflow
Reusesa group of stepsa whole job
uses: atstep level (inside a job)job level (where runs-on goes)
Declared byruns: using: compositeon: workflow_call
Inputsinputs: (untyped ok)inputs: must be typed
Best forshare steps within a jobshare a whole stage, esp. cross-repo

Composite action

yaml · actions/setup/action.ymlruns:
  using: composite
  steps:
    - uses: actions/setup-python@v5
      with: { python-version: ${{ inputs.python-version }} }
    - uses: actions/cache@v4
      ...
    - shell: bash          # MUST declare a shell
      run: pip install -r requirements.txt

Reusable workflow

yaml · reusable-tests.ymlon:
  workflow_call:
    inputs:
      python-version: { required: true, type: string }
    secrets:
      MY_TOKEN: { required: false }
jobs:
  run-tests:
    steps:
      - uses: actions/checkout@v4
      - uses: ./.github/actions/setup   # composite!

The caller (ci.yml test job)

yamltest:
  needs: lint
  strategy: { matrix: { python-version: ["3.9","3.10","3.11","3.12"] } }
  uses: ./.github/workflows/reusable-tests.yml   # where runs-on would go
  with: { python-version: ${{ matrix.python-version }} }
  secrets: inherit                              # forward repo secrets
Shape gotchaA caller job puts uses: where runs-on: would go and may NOT also have runs-on: or steps:. It MAY keep needs:, strategy.matrix, with:, secrets:. Mixing uses: with steps: is invalid.
Secrets don't auto-flowInto a reusable workflow you must pass them — declare under secrets: or use secrets: inherit to forward all at once.
Composite needs checkout firstA local action (uses: ./...) is repo files, so the caller must checkout before it — a composite can't check out the repo that contains it (chicken-and-egg). So no checkout step lives inside it.
★ Soundbite

Composite action = reuse steps; reusable workflow = reuse a job/stage, callable across repos. Edit once, propagate everywhere.

★Cross-cutting themes

If an interviewer probes why, these four ideas explain almost every decision in the pipeline.

1 · Fresh-VM isolation

Each job is a clean machine. Jobs repeat checkout/setup; sharing files between jobs needs artifacts. This is why matrices, needs, and the artifact handoff exist.

2 · Cache the inputs, not the outputs

pip downloads (Lab 5) and the Docker deps layer (Lab 7) both cache dependencies keyed by a content hash. Same theme at two altitudes; staleness is the danger.

3 · Gates everywhere

PR checks gate the merge; needs gates jobs; if gates by branch; environments gate by human approval. Layered gates = safe delivery.

4 · Run only what's needed

Path filters, schedules, fail-fast, and needs-skipping all minimise wasted runner time without weakening the gate.

Q&ARapid-fire interview Q&A

Difference between a step and a job?
A job runs on its own fresh VM; jobs run in parallel by default and can be chained with needs. Steps are ordered commands inside one job sharing that VM and its files.
How do you share a file between two jobs?
Upload it as an artifact in job A (upload-artifact) and download it in job B (download-artifact) — because each job is an isolated VM, nothing carries over automatically.
What does fail-fast: false do?
Stops a matrix from cancelling sibling jobs when one fails, so you see every version that breaks in a single run.
Why must matrix artifacts have unique names?
upload-artifact@v4 rejects two artifacts sharing a name in one run (409). Suffix with ${{ matrix.python-version }}. (v3 silently merged them.)
What makes a cache correct (not stale)?
A content fingerprint in the key — hashFiles('requirements.txt'). Change deps → key changes → fresh cache. A key with no content hash is a bug.
When does a cache get saved?
Automatically in a hidden post-job step, and only on a miss. You never write a save step; place the cache step before install.
Cron timezone & which branch?
Always UTC, no timezone field. schedule runs only on the default branch, best-effort timing, auto-disabled after 60 days idle.
The "skipped required check" deadlock?
A path-filtered workflow set as a required check shows skipped on PRs that don't touch those paths; a required check that never passes blocks the PR forever. Fix with a path-aware pattern or an always-green placeholder.
Image vs container?
An image is the built, frozen package (Dockerfile output). A container is a running instance of it. One image → many containers.
Why 0.0.0.0 in the Flask app?
127.0.0.1 only listens inside the container; 0.0.0.0 lets the mapped port be reached from outside (your machine / the CI smoke test).
Connecting to a services: container — which host?
Steps on the runner VM → localhost (with a port mapping). A job running inside a container → use the service id (postgres) as the hostname.
Three guards on a safe deploy job?
needs: test (green), if: github.ref == 'refs/heads/main' (trunk), environment: production (human approval).
Composite action vs reusable workflow?
Composite reuses steps (step-level uses, using: composite); reusable workflow reuses a whole job (job-level uses, on: workflow_call, typed inputs, callable cross-repo).
Do secrets flow into a reusable workflow automatically?
No. Pass them explicitly under secrets: or use secrets: inherit to forward all repo secrets.
Why --self-contained-html?
It inlines CSS/JS so a downloaded single report.html renders correctly; otherwise pytest-html splits assets into a folder and the lone file looks broken.
Why does if: always() matter on uploads?
Steps are skipped after an earlier failure by default; always() forces the report upload to run — and a failed build is exactly when you need the report.
